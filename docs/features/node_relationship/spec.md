# 詳細仕様：ノード間の関係性と順序の定義

## 1. 概要
LogiDo におけるノード追加時の順序、およびそれらが持つ論理的関係性を定義する。これにより、ユーザーとAIが共通の認識でタスクを分解・整理できるようにする。

## 2. 3つの主要な関係性

### 2.1 兄弟関係 (Sibling Order)
- **定義**: 同一の親ノードを持つノード同士の関係。
- **決定要因**: `order` プロパティの値（追加順にインクリメント）。
- **論理的意味**: 水平的な実行順序。Step 1 -> Step 2 の遷移。
- **Flow View での表現**: 左から右（または上から下）への直接的な接続。

### 2.2 親子関係 (Hierarchical Dependency)
- **定義**: 階層構造における「親」と「子」の関係。
- **論理的意味**: 
    - 親 = 目標（Goal）または戦略（Strategy）。
    - 子 = その達成に必要な構成要素、または具体的なアクション（Action）。
- **実行順序**: ボトムアップ。子は親の前提条件であり、子が完了することで親の進捗が更新される。

### 2.3 明示的依存関係 (Explicit Dependency)
- **定義**: `dependsOn` プロパティによる、階層を跨ぐ可能性のある制約。
- **論理的意味**: 「Aが完了しなければBを開始できない」という強制的制約。
- **優先順位**: `order` による並び順よりも優先される実行条件。

## 3. 追加時の自動推論ルール
1. **連続追加**: ユーザーが連続してノードを追加する場合、システムはそれらを「一連のプロセス（Step N, N+1...）」として扱う。
2. **分解追加**: 既存ノードの子として追加する場合、システムはそれを「既存ノードを達成するための前提条件」として扱う。

## 4. データの優先順位
AIがノードを解釈・提案する際は、以下の優先順位で関係性を考慮する：
1. `parentId` (階層)
2. `dependsOn` (強制制約)
3. `order` (推奨順序)
