# LogiDo 開発・ディレクトリ構造ルール

## 1. ディレクトリ構造

```text
src/
├── assets/         # 静的ファイル（画像、SVG等）
├── components/     # UIコンポーネント
│   ├── common/     # 汎用部品（Button, Checkbox, Icon等）
│   ├── layout/     # アプリの枠組み（Header, Sidebar, Shell等）
│   └── features/   # 特定機能に紐づくコンポーネント
│       ├── list/   # List View 関連
│       ├── tree/   # Tree View 関連
│       └── inspector/ # Why/How インスペクター関連
├── hooks/          # カスタムフック（状態管理、副作用、イベントハンドラ）
├── logic/          # 純粋なビジネスロジック（React非依存、計算、変換）
├── styles/         # グローバルスタイル、デザインシステム変数
└── utils/          # 汎用ユーティリティ（日付操作、UUID等）
```

## 2. 開発ルール

### 2.1 ロジックの分離 (Separation of Concerns)
- 複雑な計算（進捗のロールアップ等）は必ず `src/logic/` に関数として記述する。
- React コンポーネント内にはビジネスロジックを直接書かず、`hooks` または `logic` を通じて呼び出す。

### 2.2 デザインシステム
- `src/styles/index.css`（または `variables.css`）で定義された CSS 変数を優先的に使用する。
- マジックナンバー（直書きのpx等）を避け、定義されたスペーシングルールに従う。

### 2.3 コンポーネント設計
- 原則として 1 ファイル 1 コンポーネントとする。
- コンポーネント固有のスタイルは、同ディレクトリ内に `ComponentName.css` として作成する。

### 2.4 ネーミングコンベンション
- **コンポーネント**: パスカルケース (`TodoItem.jsx`)
- **フック**: キャメルケース (`useTodoTree.js`)
- **ロジック・ユーティリティ**: キャメルケース (`treeLogic.js`)
- **CSSクラス名**: ケバブケース (`todo-item-container`)

## 3. ステート管理方針
- プロジェクト全体のツリーデータは `useTodoTree.js` で一元管理し、必要なコンポーネントへ Props または Context で分配する。
- ローカルなUI状態（メニューの開閉等）は各コンポーネントで管理する。

## 4. AIアシスタント向け開発ガイドライン (AI Reference)

AIアシスタント（Antigravity）は、コードの生成・修正時に以下のチェックリストを遵守すること。

### 4.1 新規ファイル作成時のチェック
1.  **ビジネスロジックか？** → `src/logic/` へ配置（React依存を排除）。
2.  **汎用的な部品か？** → `src/components/common/` へ配置。
3.  **特定のビュー機能か？** → `src/components/features/[feature_name]/` へ配置。
4.  **状態管理や副作用を伴うか？** → `src/hooks/` へ配置。

### 4.2 スタイルの適用
- 新しいコンポーネントを作成する際は、必ず対応する `.css` ファイルを同じディレクトリに作成し、コンポーネント内でインポートする。
- グローバルな変数は `src/styles/index.css` から参照する。

### 4.3 ファイル参照
- 仕様やルールに疑義がある場合は、常に `docs/` 内の各仕様書を再読し、定義された用語（Goal, Strategy, Action等）を正しく使用する。

### 4.4 Git 運用
- 修正作業を開始する際は、必ず `docs/git_workflow.md` に従い、新しい作業用ブランチを作成してから着手する。

### 4.5 仕様書の同期 (Specification-First)
- **修正前の更新**: コードを修正する前に、リクエスト内容に応じて関連する仕様書（`rough_spec.md` や各詳細ファイル）を必ず更新・加筆すること。
- **常に最新化**: `docs/` 内のドキュメントは常に現在の実装および設計意図と完全に一致している必要がある。仕様の変更を伴う修正の場合、ドキュメントの修正もコミットに含めること。
