# LogiDo 開発運用ガイドライン (DEVELOPMENT_GUIDELINES)

> [!IMPORTANT]
> AIアシスタント（Antigravity）は、セッション開始時に必ず本ファイルを最優先で読み込み、プロジェクトの全運用ルールと設計思想を把握すること。

## 1. ワークフローの鉄則 (Workflow Rules)

### 1.1 作業開始前：ブランチの作成
- **必ず新しいブランチを作成すること**。`main` ブランチで直接作業をしてはならない。
- 命名規則: `feature/[機能名]` または `fix/[バグ名]`
- 手順: `git checkout -b feature/xxx`

### 1.2 作業中：進捗の記録
- `docs/REVISIONS.md` の対象リクエストのステータスを `[IN_PROGRESS]` に変更してから作業を開始する。

### 1.3 作業完了後：ドキュメントの同期とマージ
- **仕様書の最終更新**: タスクの完了（`[DONE]` 化）の直前に、必ず実装した内容（数値、デザイン、挙動）を `docs/specs/` 内の該当ファイルに反映し、最新の状態に保つこと。
- **Why（なぜ）の記録**: 単なる変更内容だけでなく、ユーザーとの対話で決まった背景や意図を仕様書に書き残すこと。
- **報告とマージ**: 作業ブランチ名を報告し、ユーザーの承認を得てから `main` へマージすること。

## 2. Git 運用基準

### 2.1 コミットメッセージ
以下のプレフィックスを必ず使用する。
- `feat:` 新機能 / `fix:` バグ修正 / `docs:` ドキュメントのみ / `style:` 見た目・フォーマット
- 例: `feat: implement enclosed flow with hierarchical padding`

## 3. 重要設計思想 (Core Philosophies)

### 3.1 Pure Container Flow Model (フロー表示の憲法)
- **包含構造**: 親ノードは箱ではなく、子を包む「フレーム（器）」として描画する。
- **リーフ優先**: フローの道筋はアクション（リーフノード）のみで構成する。
- **進捗同期**: 親フレームの境界線は常に子の進捗率と同期させる。

### 3.2 UI/UX ポリシー
- **リッチな質感**: グラスモフィズム、透過、アニメーションを多用し、プレミアムな体験を維持する。
- **ユーザーカスタマイズ**: 重要なレイアウト数値（余白等）は、可能な限り設定パネル（スライダー）で調整可能にする。

### 4. 連鎖修正プロトコル (Cascading Update Protocol)
ドキュメントの記述を一部修正した場合、AIは以下の手順で**連鎖的な整合性チェック**を自動的に実行すること。
1.  **キーワードの抽出**: 修正した核心的なキーワード（例：「パディング」「Pure Container」等）を特定する。
2.  **グローバル検索**: `docs/` ディレクトリ全体に対し、特定したキーワードで `grep` 検索を実行する。
3.  **不整合の解消**: 検索結果に現れた他のファイルを確認し、新方針と矛盾する古い記述があれば、それらも同時に更新する。
    - *目的*: 「Aファイルでは仕様が変わっているが、Bファイルには古い記述が残っている」という情報の断片化を完全に防ぐ。

## 5. ドキュメント構造の優先順位
AIは以下の順序で情報を参照すること。
1.  `docs/DEVELOPMENT_GUIDELINES.md` (本ファイル：運用ルール)
2.  `docs/REVISIONS.md` (現在のタスク・TODO)
3.  `docs/specs/` (各機能の正しい仕様)
4.  `docs/core/architecture.md` (システム全体設計)
