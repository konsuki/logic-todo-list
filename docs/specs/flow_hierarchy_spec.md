# 詳細仕様：包含型フロー表示 (Enclosed Flow - Pure Container Model)

## 1. 目的
親ノードを独立した「箱」として並列に置くことを廃止し、親を「子を包む器」として定義することで、論理的な包含関係と進捗の積み上げを純粋に表現する。

## 2. デザイン仕様

### 2.1 包含フレーム（親ノードの代替）
- **非並列化**: 子を持つ親ノード（Goal/Strategy）は、フロー上に独立したノード（箱）としては描画しない。
- **フレーム・アイデンティティ**: 
    - 親ノードは、その子ノード群を囲む「包含フレーム（コンテナ）」の境界線、背景、およびラベルとして表現される。
    - フレームの左上（または上部）に親ノードのタイトルを表示。
- **進捗の集約**: 親ノードの進捗率は、フレームの枠線の発光度合いやゲージとして表現し、「子が終われば器が満たされる」ことを示す。

### 2.2 子ノード（実行ユニット）
- **描画**: 子を持たないリーフノード（Action等）のみを、フロー上の実体のある「箱」として描画する。
- **配置**: フレーム内に `order` に従って順次配置。

### 2.3 接続ロジック (Connector)
- **直接接続**: 接続線は、論理的な実行順序に従い、アクション（箱）から次のアクション（箱）へ直接繋ぐ。
- **境界越え**: グループ（親）を跨ぐ場合も、前グループの最後のアクションから次グループの最初のアクションへ繋ぐ。これにより、構造的な「枠」を越えて進んでいく感覚を醸成する。

## 3. 実装の技術的アプローチ
- **フィルタリング**: `displayNodes` を生成する際、`children` を持つノードをノード描画リストから除外し、代わりに `enclosures` リストへ登録する。
- **動的サイジング**: 包含フレームのサイズは、内部に含まれる子ノード（箱）の配置範囲（Bounding Box）に基づいて動的に計算する。
- **最外郭の扱い**: `Goal` ノードなど、全ての親であるノードは「ビュー全体を緩やかに囲むフレーム」として表現することを検討。
