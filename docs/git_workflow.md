# LogiDo Git 運用ガイドライン

## 1. ブランチ戦略

| ブランチ名 | 役割 |
| :--- | :--- |
| `main` | **安定版**。常に正常に動作し、デプロイ可能なコードを保持する。 |
| `feature/[機能名]` | **新機能追加用**。`main` から分岐し、機能が完成したら `main` へマージする。 |
| `fix/[バグ名]` | **バグ修正用**。`main` から分岐し、修正が完了したら `main` へマージする。 |

## 2. 修正ワークフロー (AI連動)

1.  **計画フェーズ**:
    - `docs/REVISIONS.md` にリクエストを記載。
    - AIが `docs/implementation_plan.md` を作成・更新。
    - ユーザーが計画を承認。
2.  **作業フェーズ**:
    - **AIは作業開始前に必ずブランチを切る**。
        - `git checkout -b feature/xxx` または `git checkout -b fix/xxx`
    - コード修正を実施し、ローカルでテスト（`npm run dev`）。
    - 修正が完了したらコミット。
3.  **検証・マージフェーズ**:
    - ユーザーが新ブランチでの動作を確認。
    - 問題なければ `main` へマージし、作業ブランチを削除。

## 3. コミットメッセージのルール

以下のプレフィックスを使用する：
- `feat:` 新機能
- `fix:` バグ修正
- `docs:` ドキュメントのみの変更
- `refactor:` リファクタリング（機能変更なし）
- `style:` コードスタイルのみの変更（見た目、フォーマット等）

例: `feat: add achievement confetti animation`

## 4. AIアシスタント（Antigravity）への指示
- 修正作業を命じられた際、まず現在のブランチが `main` であることを確認し、新しい作業用ブランチを作成すること。
- 作業完了後、どのブランチで作業を行ったかを明確に報告すること。
