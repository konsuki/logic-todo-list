# Edit/Fix/Update Workflow

---
description: fix/modify/update/patch 作業時の標準ワークフロー（ガイドライン準拠）
---

## 1. 作業開始フェーズ
1. **ブランチの作成**
    - `git checkout -b feature/[機能名]` または `fix/[バグ名]` で作業用ブランチを作成する。
2. **進捗ステータスの更新**
    - `docs/REVISIONS.md` を開き、対象リクエストのステータスを `[IN_PROGRESS]` に変更する。

## 2. 実装・修正フェーズ
1. **仕様詳細の作成/更新**
    - 実装・修正に入る前に、`docs/specs/` 内に詳細仕様を記したmdファイルを作成（または既存ファイルを更新）し、実装のゴールを明確にする。
2. **実装手順（プラン）の作成と承認**
    - 実装・修正に入る前に、`docs/plans/` 内に具体的な作業手順を記したmdファイルを作成し、迷いのない実行プランを確定させる。
    - **作成完了後、そのプランファイルをタブで開き、ユーザーに実装開始の可否を確認する。**
    - 確認の際は、プランの内容を提示した上で「実装を開始してよろしいでしょうか？」と問いかけ、承認を待つ。
3. **コードの変更**
    - 確定した仕様とプランに基づき、必要な実装、バグ修正、またはアップデートを行う。
4. **連鎖修正プロトコル (Cascading Update Protocol)**
    - 修正した内容に関連するキーワードを特定し、`docs/` ディレクトリ全体を `grep` 検索する。
    - 不整合が発生している古い記述をすべて最新の状態に更新する。

## 3. 完了・同期フェーズ
1. **仕様書の同期**
    - `docs/specs/` 内の該当する仕様書を更新し、実際の実装内容や決定事項を反映させる。
2. **履歴の記録**
    - `docs/REVISIONS.md` の最上部に、`- [x] MM-DD: タイトル (@関連仕様書)` の形式で1行追記する。
3. **コミット**
    - `feat:`, `fix:`, `docs:`, `style:` などのプレフィックスを使用し、作業内容をコミットする。
4. **報告とマージのリクエスト**
    - 作業ブランチ名をユーザーに報告し、承認を得てから `main` ブランチへマージすることを提案する。
